# 设计决策记录

## 决策 18: Admin 后台改为 `/admin` 根入口下的分组路由 + 页面级按需加载

**日期**: 2026-06-16

**背景**: 原管理后台由 `frontend/pages/admin.js` 单文件承载全部面板，文件超过 3000 行，首屏会并发请求统计、AI、支付、备份、四象限、因子等多组接口。管理员即使只查看某一块，也要承担整页加载、滚动定位和移动端长页面操作成本。

**决策**:
1. 保留 `/admin` 作为总入口，但将后台拆为 `/admin`、`/admin/data`、`/admin/ai`、`/admin/ops` 四个前端路由。
2. 引入 `AdminShell` 作为统一壳层，负责 session 校验、退出登录、PC 左侧导航、移动端抽屉导航，以及旧 `?tab=` 参数到新路由的兼容跳转。
3. 各业务面板继续复用原有后端接口，不新增后端聚合 API；按页面分组组合现有请求。
4. 数据请求改为页面级按需加载，不再在总览页预取 AI、支付、备份、四象限、因子等非当前页面数据。
5. 原超长 `admin.js` 只保留总览路由入口，具体面板拆到 `frontend/components/admin/AdminSections.js`，路由与导航配置独立到 `AdminShell.js` / `navigation.js`。

**原因**:
1. 页面拆分比继续在单页内堆 tab 更容易建立稳定的信息架构，也更符合移动端操作路径。
2. 保持后端接口不变，可以把本次重构范围控制在前端，同时立刻获得可维护性和性能收益。
3. 把导航壳层、路由入口、面板实现分开后，后续新增 admin 分组时不需要再修改一个 3000+ 行巨型文件。

**替代方案**:
- 继续保留单页，仅增加页内 tab：否决，仍会首屏预拉所有数据，且移动端仍然面临超长滚动与局部状态耦合。
- 直接引入更细粒度权限模型并同时重做后台：否决，当前只有超级管理员一个角色，额外权限设计只会增加复杂度。
- 新增单一后端聚合接口供整个 admin 消费：否决，会把本已松耦合的后台能力重新绑成大接口，放大回归面。

**影响**:
- `frontend/pages/_app.js` 需要把 admin 独立布局判断扩大到所有 `/admin*` 路径。
- 管理后台测试需要从“检查单页内容”转为“检查 shell、分组路由、页面边界和按需加载约束”。
- 后续若新增 admin 分组，应优先新增独立路由和页面级请求清单，而不是回退到单页堆叠。

## 决策 16: `/live-trading` 升级为指数卡片化总览页，趋势图先走前端占位序列

**日期**: 2026-06-13

**背景**: `live-trading` 在导航重构后只保留市场指数，一期实现仅以单行文本列出 A 股与港股指数。页面信息层级过薄，既无法表达市场趋势，也不利于继续承接更多指数观察维度。当前 `/api/live/market/overview` 只稳定提供指数点位与涨跌信息，尚未提供可直接消费的指数趋势序列。

**决策**:
1. `/live-trading` 继续复用现有两路 overview 接口，不新增后端趋势接口作为本次前置条件。
2. 页面结构升级为 `Hero 总览 + 核心指数卡片 + 扩展指数观察 + 市场观察摘要` 四层 dashboard。
3. 首屏核心指数固定为 6 张：上证指数、深证成指、创业板指、恒生指数、恒生中国企业指数、恒生科技指数。
4. 扩展指数观察优先按前端可识别代码承接沪深300、科创50、上证50、中证500；若数据源未返回，则明确展示推荐补充提示，而不是静默缺失。
5. 在后端尚未提供真实趋势序列前，卡片内迷你图先用“基于最新点位与涨跌幅推导的 7 点占位序列”渲染，目标是提供趋势感与卡片视觉支撑，不把它包装成专业分时图。

**原因**:
1. 当前用户诉求是先把页面从“单薄指数列表”提升为“可读的市场总览”，前端卡片化与摘要化收益最高、回归面最小。
2. 若等待后端补齐趋势接口再改 UI，会继续拖延页面可用性；而占位趋势图足以满足当前“丰富页面、强化趋势感”的目标。
3. 把补充指数收敛在固定代码白名单里，可以避免页面因为接口返回顺序漂移而出现布局抖动。

**替代方案**:
- 先新增后端 `index-trends` 接口再动前端：否决，本次目标是快速完成页面升级，不应扩大到后端联动改造。
- 继续只展示文本列表、额外堆说明文案：否决，不能解决趋势感缺失和页面单薄问题。
- 直接接入完整专业图表组件：否决，当前首页定位是总览，不需要引入高交互成本和额外性能负担。

**影响**:
- `frontend/pages/live-trading.js` 需要抽出指数标准化、卡片分组、市场摘要与占位趋势图构造逻辑。
- 页面测试不再只校验“仍然只有市场 overview 数据”，还要覆盖核心/扩展卡片分组与新增指数映射。
- 后续若后端补充真实趋势字段，应优先替换 `buildTrendSeries(...)`，而不是重写卡片布局。

## 决策 17: `/portfolio-tracking` 改为按市场分区的四组合平铺总览

**日期**: 2026-06-14

**背景**: `portfolio-tracking` 页面当前只有 4 个固定组合（A股-A / A股-B / 港股-A / 港股-B），但旧版仍采用“市场 tab + 组合 tab”的双层切换结构。用户在比较不同市场与不同组合时需要频繁切换，且滚动后容易丢失“当前看到的是哪个市场”的上下文。

**决策**:
1. 页面取消 `selectedExchange` / `selectedVariant` 双 tab 交互，改为 A股、港股两个 section 常驻展示。
2. 每个市场 section 固定展示 2 张组合卡片（A / B），形成“市场分区 + 组合卡片”的二维总览结构。
3. A股与港股使用不同视觉语义：A股采用暖色 accent，港股采用冷色 accent，同时在 section 和卡片头重复展示市场 badge，强化市场识别。
4. 单卡信息密度从“完整详情页”压缩为“4 个核心指标 + 3 个次级指标 + 迷你曲线 + 最多 4 条成分股 + 折叠式调仓”，优先保障四卡并排时的扫读效率，并减少额外说明文案。
5. 当单个组合缺失时保留占位卡，避免市场分区与 A/B 卡位因后端数据波动而发生布局跳动。

**原因**:
1. 该页面信息域单一，核心任务就是比较 4 个组合表现，不需要用 tab 隐藏内容。
2. 先按市场分区，再在区内横向比较 A/B，比 4 卡完全混排更符合用户认知，也更容易一眼区分 A股和港股。
3. 单卡降密度能把比较动作前置到首屏，同时保留成分股和调仓明细作为二级信息，不牺牲信息完整性。

**替代方案**:
- 保留 tab，仅增加快捷切换：否决，仍然无法解决“同屏对比 4 个组合”的核心问题。
- 直接 2x2 混排 4 张卡，不做市场 section：否决，市场边界太弱，A股/港股识别不够稳。
- 每张卡继续保留完整 7 指标 + 长图表 + 全量持仓：否决，四卡平铺下信息过载，难以快速扫读。

**影响**:
- `frontend/components/RankingPortfolioPanel.js` 需要引入市场配置、视图模型和 section/card 结构拆分，替换旧的选中态详情布局。
- 组件测试需要从校验 tab 状态切换改为校验市场分区、四卡平铺与颜色语义。
- 后续若新增市场或组合，应继续沿用“market section -> fixed card slots” 模式，而不是重新引入 tab。

## 2026-06-16 AI Picker 放开 LLM 输出 token 限制

- 决策：`callLLM` 不再设置 `max_tokens`，完全由 provider 默认策略决定输出上限；同时将 HTTP timeout 从 90s 提升至 240s。
- 原因：当候选池 60+ 条 × 每条 20+ 字段 + 每只股票要求 3-5 句中文理由时，完整 JSON 输出超过 4096 tokens，导致稳定触发 `finish_reason=length`。DeepSeek 支持 1M 上下文，provider 本身有充足能力；人为限制 `max_tokens` 是根因。
- 替代方案：收缩 schema 字段、压缩 prompt 候选数量、压缩每只股票输出约束。
- 不选原因：产品定位要求完整中文理由和全字段输出；这是一天只运行一次的低频服务，token 成本不是约束。
- 风险与代价：若 provider 仍有默认上限导致截断，错误信息现在更具可观测性（含 `completion_tokens` / `prompt_chars`），便于排查；HTTP timeout 延长增加了单次请求的最大等待时间，已在 admin 页面通过异步生成+状态查询规避用户感知。

## 2026-06-15 AI Picker 技术快照软增强
- 决策：AI Picker 中技术快照从“硬门槛”改为“软增强”；候选股缺少技术快照时仍保留，只将相关技术字段置空，并在 prompt 中显式标注技术信息缺失。
- 原因：技术快照依赖外部日线接口，单点失败不应导致整个候选池被清空；当前真实主链路是因子召回，技术面只是增强信息。
- 替代方案：继续把技术快照作为硬过滤条件；技术快照失败时直接报错阻断生成。
- 不选原因：硬过滤会把“增强失败”误表现为“候选池为空”，导致生成可用性过低，也不符合因子优先的产品定位。
- 风险与代价：部分推荐会缺少技术面细节，需通过 prompt 明示并在理由中说明“技术面信息不足”；同时仍需保留日志以便后续排查技术快照链路。

## 2026-06-14 AI Picker P0
- 决策：A 股 AI 选股采用 `factorlab` 候选池 + LLM 二次决策，而不是纯 prompt 直出。
- 原因：减少幻觉，允许理由中引用真实因子分值，并复用现有日更因子资产。
- 替代方案：纯 prompt 选股、港股同时接入。
- 不选原因：纯 prompt 缺乏真实数据约束；港股因子链路尚未就绪。
- 风险与代价：结果质量仍依赖 prompt 与 post-validation；每日结果表当前只保存最终 JSON，不保存更细中间候选日志。
