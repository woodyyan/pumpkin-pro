## 决策 26: 模拟组合事实表 Admin 必须同时暴露流程阶段与同步结果

**日期**: 2026-07-01

**背景**: `/portfolio-tracking` 已切到新事实表链路，但本地排查发现部分组合只有 seeded baseline，`portfolio_position` / `portfolio_trade` / `portfolio_metrics` 为空。前置开盘价和 legacy 结果表都可能已有数据，但页面仍为空。原 admin 面板只展示缺开盘价和状态，且「补齐建仓开盘价」容易被误解为会生成持仓。

**决策**:
1. Admin 状态表必须区分「前置价格是否就绪」与「事实表是否已生成」，新增事实表行数、`baseline_only`、`can_sync`、下一段 `signal_date -> trade_date` 和 `action_hint`。
2. Admin 文案明确固定运维顺序：先补开盘价、再同步事实表、最后验证一致性；补开盘价不生成持仓/调仓。
3. `SyncSimPortfolios` 不再只返回 error，而是返回 per-portfolio 同步摘要，包括锚点、最新信号、生成估值日数量、最后生成日期和阻断原因。
4. 四象限 bulk-save 成功后的自动同步继续异步执行，但日志必须输出本轮生成估值日数量，避免 silent no-op。

**原因**:
1. seeded baseline 是合法中间态，但对运维来说必须可见，否则会误判为页面或前端问题。
2. 开盘价补齐只是前置动作，事实表生成是独立动作；把两者混在一个状态里会导致误操作。
3. per-portfolio 摘要能快速定位是哪个组合未推进、为什么未推进，避免只看到全局成功。

**替代方案**:
- 让「补齐建仓开盘价」自动同步事实表：否决。补价和写事实表是不同副作用，合并会降低可排障性。
- 只在前端加说明文案，不改后端响应：否决。缺少 per-portfolio 结果时，自动同步和手动同步仍不可观测。
- 新增独立同步日志表：暂缓。当前低频运维先用同步响应和 bulk-save 日志满足可观测性；若后续需要历史审计再持久化。

**风险与代价**:
- Admin status 查询增加少量 count 查询；当前组合固定为 4 个，成本可接受。
- 同步响应结构变更为对象摘要；现有 admin handler 测试已覆盖 `items`，前端按新结构展示。

## 决策 25: MiniChart Y 轴采用自适应 min/max + padding 而非起点归零

**日期**: 2026-07-01

**背景**: `/live-trading` 市场行情页的单因子指数卡与大盘指数卡共用 `MiniChart`（Canvas 迷你图）。原实现 Y 轴固定从 0 缩放到 `max(values)`，导致因子指数 NAV（0.98~1.08）真实波动只占画布约 8%、上证综指（3100~3250）约 4%，所有曲线「看起来都很平」，无法反映真实波动。

**决策**:
1. Y 轴从固定 [0, max] 改为数据 [dataMin − pad, dataMax + pad] 自适应，`pad = max(range*0.15, absMax*0.01)`。
2. 持平 / 极小波动（range/absMax < 0.1%）时以数据中点为中心构建 ±0.5% 虚拟范围；全 0 退化为 [0,1]。
3. 刻度值显示真实数值（不归零、不缩放），精度按量级自适应（≥100 取整，否则 2 位）。
4. 缩放逻辑抽成纯函数 `lib/mini-chart-scale.js`（`buildYAxisScale` 等），便于单测。
5. `MiniChart` 新增可选 props `yAxisMode`（默认 `auto`）/ `valuePrecision` / `baselineValue`，现有调用零改动即生效。
6. 保留 `zero-based` 模式给未来柱状图等必须从 0 起算的场景。

**原因**:
1. 卡片头部已展示绝对 NAV/点位与涨跌幅百分比，曲线只承担「辅助呈现形状与拐点」职责，不必再承担跨指数对比。
2. 自适应 min/max 是 Bloomberg、东方财富、同花顺等金融图表的标准做法，保留绝对值语义，符合用户直觉。
3. 改动集中在 MiniChart 内部 + 一个纯函数文件，不触及数据层与卡片布局，风险最小。

**替代方案**:
- 起点归零（所有点减起点值，Y 轴表相对变化）：否决。改变曲线语义，与卡片上方绝对值/涨跌幅文本割裂；起点本身随机，归零后曲线形状对起点敏感。
- 百分比归一化（起点=100%）：否决。完全丢失绝对值，与已有涨跌幅文本重复。
- 引入 Recharts/ECharts：否决。当前 Canvas 实现仅 130+ 行，引入第三方库带来 50KB+ 包体积与主题适配成本，收益不匹配。

**风险与代价**:
- 自适应会视觉放大波动，用户可能误以为「涨很多」。缓解：Y 轴刻度显示真实数值，卡片头部涨跌幅文本为权威信息。
- 跨卡片对比失真（每张卡 Y 轴范围不同）。缓解：sparkline 本就不适合精确跨卡对比，卡片头部涨跌幅是跨卡对比权威依据。
- `MiniChart` 硬编码 `rgba(255,255,255,...)` 文字/网格色未顺带修复，浅色主题下 Y 轴可读性差，作为独立后续项。

**关联文档**: `.ai/specs/mini-chart-yaxis/`

## 决策 24: 模拟组合事实表链路新增补开盘价补偿与缺口可观测性

**日期**: 2026-07-01

**背景**: `/portfolio-tracking` 四个模拟组合因建仓开盘价填充链路断裂（09:25 worker 错过窗口或 market price 行未先生成），导致事实表只停留在 seeded baseline，页面永久显示「等待下一交易日开盘价」且无任何失败状态暴露。

**决策**:
1. 新增 `BackfillSimPortfolioOpenPrices` service 方法，从日线 open 精确补齐 T+1 建仓开盘价，支持 `latest_only` 轻量模式和按组合/市场筛选。
2. 新增 `POST /api/admin/portfolio-tracking/backfill-open-prices` API 与 admin 按钮「补齐建仓开盘价」，作为新口径事实表链路的独立运维入口。
3. `SyncSimPortfolios` 内部先执行一次 `backfillSimPortfolioOpenPrices(latest_only=true)` 再同步事实表，具备最小自愈能力。
4. `RecomputeSimPortfolios` 内部先执行全量 `backfillSimPortfolioOpenPrices` 再重算，避免重算时因缺价失败。
5. admin status 响应新增 `missing_open_price_count` 和 `missing_close_price_count` 字段；admin 状态表新增「缺开盘价」列，琥珀色标记缺口。
6. `VerifySimPortfolios` 新增 `missing_open_price` 校验状态，检测持仓 `buy_price <= 0` 的记录。
7. 所有运维动作互斥，同一时间只能执行一个写库操作（sync / recompute / backfill / verify）。

**原因**:
1. 补开盘价是恢复事实表链路的最小修复动作，不应每次都从头重算。
2. sync/recompute 内部自动补价能覆盖大部分自动恢复场景，但 admin 仍需独立按钮用于排障。
3. 缺口计数让运维一眼看到"缺什么"而不是只看到"pending"。
4. verify 覆盖开盘价完整性可以防止事实表写入了无效 buy_price=0 的持仓。

**替代方案**:
- 只在 sync 内部补价，不加独立按钮：否决，运维需要分步排障能力。
- 新建 `portfolio_entry_price` 独立表去 legacy 依赖：暂缓，当前复用 legacy 表成本更低，封装在 service 后面即可。
- 用异步任务替代同步执行：暂缓，当前数据量小，同步返回足够快；后续数据量变大再改异步。

**风险与代价**:
- 补开盘价依赖外部日线接口，如果接口不可用仍会 pending；admin 应提示"可能是停牌、日线源未更新或目标交易日尚未到达"。
- latest_only 模式只补最新批次，历史缺口需要手动指定范围或走 recompute。
- 所有运维动作串行化，如果某个动作卡住，其他动作会等待；但这对当前低频运维场景可接受。

## 决策 23: 单因子指数以内嵌卡片形式接入 `/live-trading`，并采用后端全量物化链路

**日期**: 2026-06-30

**背景**: 产品希望在「市场行情」页直接观察 7 个单因子指数，不希望再引入额外详情页或前端临时计算。现有仓库已经具备 `factor_scores`、`factor_daily_bars` 和市场行情 dashboard，但尚未形成可直接展示的因子指数物化结果。

**决策**:
1. 单因子指数首期不新增详情页，7 张因子指数卡直接放在 `frontend/pages/live-trading.js` 的「核心指数卡片」上方。
2. 指数口径固定为：每月首个 A 股交易日收盘后按对应因子 score 选 Top50，下一交易日生效，等权 2%，月内不调仓。
3. 所有结果必须后端物化：新增独立 `factorindex` 模块和事实表，前端只请求 `/api/live/factor-index/overview` 展示 NAV、区间收益、状态和轻量净值曲线，不在页面内做收益/NAV 计算。
4. 调度拆为两段：月度调仓默认北京时间 20:10，日级 NAV 默认北京时间 20:30；worker 启动后先做一次 catch-up，同步历史调仓和日净值。
5. 管理后台统一挂到 `/admin/data`：提供 `/api/admin/factor-index/status` 查询 worker 与物化状态，`/api/admin/factor-index/recompute` 触发异步手动补算 / 从头重建，避免在 HTTP 请求内同步执行长任务。
6. 首期只支持 A 股，默认 7 个固定因子：价值、股息率、成长、质量、动量、小市值、低波动。

**原因**:
1. 因子指数的信息密度不足以支撑独立详情页，直接放在 `/live-trading` 更符合“同屏看市场与风格”的产品目标。
2. 后端物化可以锁定历史成分股和 NAV 曲线，避免前端按当下数据临时拼装导致口径漂移。
3. 调仓与日估值拆成两个时间点，可避免“当日调仓影响当日收益”的未来函数问题，并为上游收盘数据留出缓冲。
4. 单独 `factorindex` 模块能复用 Factor Lab 数据，又不污染现有 `portfolio_daily` / `portfolio_tracking` 组合语义。

**替代方案**:
- 新增 `/live-trading/factor-index` 详情页：否决，首期价值主要是快速横向对比，跳转会拉长路径。
- 前端直接根据成分股行情实时计算 NAV：否决，无法稳定复现历史口径，也违背“用户态读接口不隐式写库/算库”的长期原则。
- 复用 `portfolio_daily` 等模拟组合事实表：否决，指数方法论与模拟组合语义不同，混用会增加后续维护成本。

**风险与代价**:
- 因子与日线数据若在 20:10/20:30 尚未就绪，worker 需要依赖重试和启动 catch-up，运维上要接受“稍晚完成但仍保持 `source_trade_date` 正确”的状态。
- 首期卡片不展示 Top50 明细或 benchmark，对专业用户解释力偏弱；后续若信息量上升，再考虑抽屉或二级详情页。
- `factor_index_rebalances` 的生效日和结束日不能用自然日推算，后续所有扩展仍必须沿用交易日链路。

## 决策 22: 四象限自动计算固定为北京时间 20:00，并按市场交易日分别跳过

**日期**: 2026-06-29

**背景**: 原 `quadrant.Worker` 固定在北京时间凌晨 02:00 触发四象限计算。虽然能规避部分数据落盘延迟，但会让 `computed_at` 与用户理解的收盘日期天然错位，也不利于后续依赖“当天收盘后结果”的其他任务链路。

**决策**:
1. 四象限自动计算改为配置驱动，默认北京时间每日 20:00 执行，配置统一放入 `config.Quadrant`，并由 `main.go` 透传给 worker。
2. worker 必须按 A 股 / 中国香港两个市场分别判断是否为交易日；某一市场休市时，只跳过该市场，不连带跳过另一市场。
3. 自动任务的交易日判断与 A 股 `source_trade_date` 解析优先使用外部 benchmark 日线确认当天收盘口径，内部 closing snapshot 只作为回退来源。
4. 管理后台与用户态文案统一改为“每日 20:00 收盘后自动计算”，不再出现“凌晨定时任务”描述。

**原因**:
1. 20:00 调度更容易让 `computed_at` 与 `source_trade_date` 保持在同一自然日，减少用户和下游任务对日期口径的误解。
2. A 股与中国香港交易日不完全一致，市场维度跳过能避免在一边休市时错误触发另一边，或把两个市场的状态揉成一个结果。
3. 外部 benchmark 日线比内部 closing snapshot 更接近“当天收盘是否已经可用”的真实口径，能降低 20:00 刚触发时因为内部数据未落盘导致的错误回退。

**替代方案**:
- 继续保留凌晨 02:00：否决，会持续制造 `computed_at` 与 `source_trade_date` 的自然日错位。
- 仅把硬编码 02:00 改成 20:00：否决，缺少配置化和按市场跳过能力，后续再改时间或拆分市场会继续返工。
- 继续只让 A 股成功后再顺带触发中国香港：否决，无法表达“港股今天休市但 A 股照常运行”这类市场差异。

**风险与代价**:
- 若 20:00 时外部日线源仍未更新到当天，系统只能回退到最近交易日；因此日志和状态必须允许出现“20:00 触发但仍使用上一交易日”的情况。
- 新增配置项后，部署环境需要同步补齐 compose / env 默认值，否则会出现环境间行为不一致。
- 管理后台前端状态需要识别 `skipped`，否则运维会误以为任务未执行。

## 决策 21: 资金星图以 `/capital-map` 站内原生页接入看板

**日期**: 2026-06-28

**背景**: 独立项目 `liquidity-layer-monitor` 已实现单页「资金星系」，基于东方财富公开行情接口展示 A 股高流动性样本的 PE、成交额、板块资金流和 PoC 估值锚。需要迁入卧龙网站，并放入「看板」分组，命名为「资金星图」，位于「市场全景」之后。

**决策**:
1. 路由固定为 `/capital-map`，导航显示「资金星图」，归属「看板」分组并排在「市场全景」之后。
2. 首期仅支持 A 股，页面公开可见，权限口径与「市场行情」「市场全景」保持一致。
3. 允许引入 `echarts` 作为前端图表依赖，以保持原页面散点图、板块资金图和 PE 分箱图能力。
4. 数据源首期继续沿用东方财富公开行情接口，但外部请求必须封装在卧龙 Go 后端 `/api/capital-map`，前端不得直接请求东方财富。
5. 保持原刷新和算法口径：前端 60 秒刷新；后端 30 秒内存缓存；PE 优先使用 PE TTM，否则回退动态 PE；PoC 按 `0 < PE <= 120`、5 倍 PE 分箱、分箱成交额最大值计算。
6. UI 必须使用卧龙语义化 Tailwind token，并根据 `useTheme().resolvedTheme` 在图表配置中切换浅色/深色配色；A 股涨跌颜色继续采用红涨绿跌。

**原因**:
1. 原页面能力高度依赖东方财富的 PE、主力净流入和板块资金字段；直接换源会破坏“刷新频率、算法和功能不变”的目标。
2. 把外部源封装在 Go 后端可复用鉴权、缓存、错误处理与未来运维观测能力，并避免前端暴露第三方接口细节。
3. ECharts 能低成本保持原交互与图表表达，避免为散点 tooltip、视觉映射和双轴图重写自研图表。

**替代方案**:
- 直接 iframe 原站：否决，会割裂导航、主题、权限和运维链路，也无法统一浅色/深色适配。
- 复用 Factor Lab 离线因子数据：否决，Factor Lab 是夜间批处理，不满足 60 秒资金观察口径。
- 直接使用现有腾讯行情源：否决，现有 live 链路不完整覆盖 PE TTM、板块主力资金和 PoC 所需字段。
- 不引入 ECharts、改用自研 SVG/Canvas：否决，迁移成本高且容易偏离原功能。

**风险与代价**:
- 东方财富公开接口不是正式商用授权数据源，页面必须保留数据口径和风险说明；生产商用前需确认授权或切换授权源。
- 外部字段可能漂移，后端需集中解析并保留缓存降级；后续如页面流量上升，应演进为 worker 定时采集 + SQLite 快照。
- 新增 ECharts 会增加前端依赖和图表 chunk，需通过动态 import 限制首屏影响。

## 决策 20: 首页跟随导航升级为“AI投研工作台”信息架构

**日期**: 2026-06-27

**背景**: 主站导航已更新为「卧龙AI / 看板 / 跟踪 / 选股 / 更多」，产品能力也新增或强化了 AI分析、AI研报、AI选股、因子实验室、组合跟踪、持仓管理等入口。旧首页仍以「A股+港股 / 卧龙AI投研模型 / 100+指标」作为核心卖点，无法准确表达当前产品价值。

**决策**:
1. 首页主叙事从“量化工具集合”升级为“AI投研工作台”，强调「发现市场机会 → AI判断 → 因子验证 → 组合跟踪 → 持仓复盘」。
2. 首页三大核心卖点固定为「AI 投研闭环」「因子驱动选股」「组合跟踪与复盘」，不再把“市场覆盖”和“指标数量”作为第一层卖点。
3. 「快速上手」按用户任务组织，覆盖 AI分析、AI研报、AI选股、因子实验室、组合跟踪、持仓管理六条路径。
4. 「我们提供什么」按能力域分类展示：AI投研、市场与机会发现、选股与策略研究、跟踪与组合管理、账户服务。
5. 首页内容统一收敛到 `frontend/data/homepage.js`，`frontend/pages/index.js` 只负责渲染和登录 CTA 行为。
6. AI分析、AI研报、AI选股、因子排序、回测、模拟组合、交易信号相关入口必须保留显著风险提示，避免表达为收益承诺。

**原因**:

## 2026-06-16 AI Picker 放开 LLM 输出 token 限制

- 决策：`callLLM` 不再设置 `max_tokens`，完全由 provider 默认策略决定输出上限；同时将 HTTP timeout 从 90s 提升至 240s。
- 原因：当候选池 60+ 条 × 每条 20+ 字段 + 每只股票要求 3-5 句中文理由时，完整 JSON 输出超过 4096 tokens，导致稳定触发 `finish_reason=length`。DeepSeek 支持 1M 上下文，provider 本身有充足能力；人为限制 `max_tokens` 是根因。
- 替代方案：收缩 schema 字段、压缩 prompt 候选数量、压缩每只股票输出约束。
- 不选原因：产品定位要求完整中文理由和全字段输出；这是一天只运行一次的低频服务，token 成本不是约束。
- 风险与代价：若 provider 仍有默认上限导致截断，错误信息现在更具可观测性（含 `completion_tokens` / `prompt_chars`），便于排查；HTTP timeout 延长增加了单次请求的最大等待时间，已在 admin 页面通过异步生成+状态查询规避用户感知。

## 2026-06-17 Admin AI 页面保留最近一次 AI Picker 完整交互详情，但不扩展为历史全量归档
