# Design Decisions

## D-001: COS 备份上传超时与重试策略

**日期**: 2026-05-20

**决策**: 单次 COS 上传超时从 60s 调大至合理值；单文件上传失败后自动重试 1 次。

**原因**: 服务器在新加坡（4Mbps 上行），COS 在广州（ap-guangzhou），跨地域无代理。pumpkin.db ~300MB 且持续增长，60s 超时不足以完成上传（300MB / 4Mbps ≈ 600s 理论下限）。超时后 COS 同步标记为「部分同步」。

**替代方案**:
1. 分片上传（Multipart Upload）— 可断点续传，但实现复杂度高，暂不采用。
2. 压缩后上传 — 减小传输体积，但增加 CPU 开销及备份恢复复杂度，暂不采用。
3. 加大超时 + 重试 — 改动最小，先验证效果。

**风险**: 超时调大后单次同步耗时会变长；若网络中断，重试仍可能失败，但至少覆盖瞬时抖动场景。

**待观察**: 验证效果后决定是否需要引入分片上传或压缩。

## D-002: 因子实验室股息率数据源优先级

**日期**: 2026-05-21

**决策**: A 股股息率接入采用「直接股息率优先，现金分红解析补充」策略。Phase0 dividends 采集时优先写入 `factor_dividend_records.dividend_yield`，字段来源优先级为：
1. AkShare `现金分红-股息率`
2. 东方财富 `DIVIDENT_RATIO`
3. 腾讯基础面 `items.dividend_yield`
4. 若直接股息率缺失，再从 AkShare `现金分红-现金分红比例描述` 或东方财富 `IMPL_PLAN_PROFILE` / `PRETAX_BONUS_RMB` 解析 `cash_dividend_per_share`

**原因**: 探测结果显示 AkShare 和东方财富对有分红股票提供直接股息率字段；历史实现只保存 `cash_dividend_per_share` / `total_cash_dividend`，但这些字段全为空，导致 `factor_snapshots.dividend_yield`、`factor_rank_scores.dividend_yield_rank_score`、`factor_scores.dividend_yield_score` 全部为空。

**替代方案**:
1. 只解析每股现金分红再用股价估算 — 口径更可控，但覆盖和解析成本更高。
2. 只使用腾讯 `items.dividend_yield` — 接入简单，但样本中存在空值，且来源链路较间接。
3. 只使用 AkShare — 字段语义清晰，但部分股票会抛异常或无记录。

**风险**: 不同数据源的股息率口径可能不同。必须保留 `dividend_yield_source` 和 `raw_plan`，便于 Admin 诊断和后续口径校准。

**验证**: Phase0 dividends 后检查 `factor_dividend_records.dividend_yield` 非空覆盖；Phase1 后检查 `factor_snapshots.dividend_yield` 覆盖；Phase2 后检查 `factor_scores.dividend_yield_score` 覆盖。

## D-003: 因子流水线支持分阶段与修复范围运行

**日期**: 2026-05-21

**决策**: Admin 手动触发因子流水线不再只能运行完整链路，而是支持 `phase=all|phase0|phase1|phase2|phase1_phase2`，Phase0 支持 `mode=all|securities|daily-bars|index-bars|financials|dividends`，并提供字段修复 scope：`repair_missing_dividend_yield` 与 `repair_missing_operating_cash_flow`。

**原因**: 因子数据缺口经常集中在单一阶段或单一字段（例如只缺股息率或经营现金流），完整运行 Phase0→Phase1→Phase2 会重复拉取大量不相关数据，耗时且浪费外部数据源额度。

**替代方案**:
1. 继续只提供完整流水线 — 操作简单，但排障效率低且资源浪费。
2. 只提供命令行脚本 — 灵活，但会绕开 backend 容器单写 SQLite 的安全边界。
3. Admin 提供分阶段请求 — 保持单写入通道，同时提升修复效率。

**风险**: 分阶段运行会引入状态组合复杂度；需要后端校验非法组合，例如 `repair_missing_dividend_yield` 只能配合 `phase0_mode=dividends`。

**验证**: Admin 触发按钮应能分别运行 Phase0/Phase1/Phase2/Phase1+2；修复股息率后需再跑 Phase1+2 才能反映到因子得分。

## D-004: 模拟组合“当前成分股”跟随最新排行榜，收益曲线继续使用已落盘批次

**日期**: 2026-05-22

**决策**: `GET /api/quadrant/ranking-portfolio` 返回的模拟组合数据拆成两层语义：
1. 收益曲线、累计收益、最近一次调仓继续读取已落盘的 `quadrant_ranking_portfolio_results`，保持批次一致性与可回放性。
2. “当前成分股”不再直接复用该结果中的 `constituents_json`，而是基于最新排行榜请求时实时重算，并额外返回 `current_constituent_source_date` 与 `current_constituent_effective_time` 供前端展示“按 T 日收盘后榜单生成，T+1 开盘生效”。当最新排行榜晚于已落盘组合批次时，隐藏旧的“最近一次调仓”信息，避免把旧批次调仓误认为当前持仓。

**原因**: 生产环境中四象限/排行榜已更新到新交易日，而模拟组合结果表仍停留在旧批次。若 API 直接返回旧批次的 `constituents_json`，用户会看到“当前成分股”与排行榜 TOP4 不一致，误以为策略口径不同；实际上问题是离线结果刷新滞后，而用户需要的是开盘前可直接抄作业的最新名单。

**替代方案**:
1. 强制要求模拟组合落盘任务与排行榜同批次完成后才对外展示：数据最一致，但一旦离线任务卡住，整个模块持续过期。
2. 直接把 `snapshot_date` 改成最新排行榜日期：展示上看似解决，但会把旧收益批次伪装成新批次，制造更大的口径混淆。
3. 收益与当前成分股分层返回：收益保持历史批次一致，当前成分股对齐最新排行榜，前端用明确文案揭示时间语义。

**风险**: 同一响应内会同时存在“历史收益批次时间”和“当前成分股生效时间”两套时态。前端必须显式写清楚，否则用户仍会误读；同时需要接受“最近一次调仓”在最新排行榜晚于落盘批次时被临时隐藏。

**验证**: 当排行榜已更新、模拟组合结果未补算时，A 股/港股“当前成分股”应与最新排行榜规则选股一致；文案应显示“按 T 日收盘后榜单生成，T+1 开盘生效”；旧批次 `latest_rebalance` 不应继续展示为当前调仓。

## D-005: 用户侧日期统一展示 source_trade_date

**日期**: 2026-05-22

**决策**: 所有"日级收盘后产出"的策略、榜单、象限、组合，用户主界面统一展示 `source_trade_date`（真实收盘交易日），不再使用 `computed_at` 或 `snapshot_date` 充当用户口径日期。前端由统一 helper 将 `source_trade_date` 渲染为文案 `按 {date} 收盘后数据生成`。后端将 `source_trade_date` 作为批次级元数据写入结果表，A 股和港股按各自交易日历独立计算。

**原因**: 凌晨 02:00 完成的离线计算，其数据基于前一天收盘数据，但 `computed_at` 显示为计算当天日期（如 5/22），导致用户误以为这是 5/22 收盘后的数据。用 `computed_at - 1 day` 推算不可靠：周末、节假日、A股/港股交易日差异均无法通过简单减天数处理。

**替代方案**:
1. 用 `computed_at` 减 1 天推算收盘日 — 周一凌晨算出的 `computed_at - 1 = 周日`，实际收盘日应为上周五；港股台风休市同理。不可行。
2. 只修 3 个显示点，不建立全局规范 — 后续新模块仍会重复犯错。不可取。
3. 全局规则 + 统一 helper + 批次级元数据 — 改动可控，长期收益明确。采纳。

**风险**: 已有模块的 `source_trade_date` 回填需要确保历史数据与交易日历一致；若交易日历数据本身有误，所有依赖模块的日期都会错。需保证交易日历服务的准确性。

**验证**: 行情看板 3 个位置（风险机会全景图、排行榜、模拟组合当前成分股）均展示 `按 {source_trade_date} 收盘后数据生成`；跨周末/节假日后首次计算，日期应为最后一个交易日而非自然日减 1。

## D-006: 因子实验室权重输入改为百分比与“草稿-已应用”手动应用模式

**日期**: 2026-05-22

**决策**: 因子实验室页面的“因子与权重”改为面向用户的百分比输入与展示，并把右侧股票列表从自动应用改为手动应用。
1. 权重输入口径从 `0-1` 小数改为 `0-100` 的百分比数值，UI 显示为 `{value}%`。
2. 允许小数百分比（例如 `12.5%`），最多保留 2 位小数；输入值范围为 `0` 到 `100`（含边界）。
3. 页面采用“双状态”模型：左侧为可编辑“草稿配置（Draft）”，右侧列表始终基于“上一次已应用配置（Applied）”渲染。
4. 用户修改权重后，必须点击左侧底部按钮 `启动计算` 才会应用草稿并触发右侧列表更新。
5. 首屏仍自动展示一版“默认 7 因子等权结果”（即自动产生一次默认 Applied）。
6. 当草稿未重新应用时，右侧排序与翻页继续可用，但必须严格基于当前 Applied 的结果与参数，不跟随 Draft。
7. `重置` 按钮语义固定为“仅重置左侧草稿”，不影响右侧当前展示的 Applied。
8. `启动计算` 按钮文案固定为“启动计算”，且按钮旁不增加任何辅助说明。

**原因**:
1. 百分比输入更符合用户直觉，降低把 `30%` 误写成 `0.3` 的心智负担与出错率。
2. “草稿-已应用”解耦可避免频繁自动触发计算带来的抖动，提升可控性，并保持列表交互（排序/翻页）稳定。
3. 首屏默认结果保留，确保页面打开即有可用内容，减少空状态。

**替代方案**:
1. 继续使用 `0-1` 小数权重：实现成本低，但更容易输入错误且不符合习惯。
2. 仍然自动应用（onChange / debounce）：交互“看起来更即时”，但计算触发频繁，且用户在编辑中间态会导致列表不断变化。
3. 保留 `启动计算` 但增加解释文案/tooltip：可降低疑惑，但用户已确认不需要额外说明，且会增加 UI 噪音。
4. 重置同时重置 Draft 与 Applied：简单，但会破坏“右侧稳定基于已应用配置”的预期，并导致用户丢失可回退的已应用结果。

**风险**:
1. Draft 与 Applied 状态并存会引入一致性边界：必须确保右侧所有查询条件（含排序、分页）都绑定 Applied，而不是意外混用 Draft。
2. 百分比到内部权重的换算与取整需要一致策略（最多 2 位小数），避免前后端口径漂移；权重总和是否需要约束应沿用既有规则，不在本次引入新约束。

**验证**:
1. 输入 `30` 显示为 `30%` 并按 `0.30`（或等价内部口径）参与计算；输入 `12.5` 显示为 `12.5%`，最多两位小数。
2. 未点击 `启动计算` 前：右侧列表不更新，但排序/分页可正常操作且结果只反映上一次 Applied。
3. 点击 `启动计算` 后：Applied 更新为当前 Draft，右侧列表随之更新。
4. 点击 `重置`：仅左侧草稿恢复默认，不改变右侧当前展示结果。
5. 首屏进入页面：自动看到默认 7 因子等权结果。


## 2026-05-25 密码重置与会话撤销
- 用户密码重置采用数据库持久化随机 token，服务端只保存 `SHA-256` 哈希，避免明文 token 落库。
- 重置成功时必须在一个数据库事务内完成：消费 token、更新密码哈希、递增 `credential_version`、撤销该用户全部 session。
- access token 追加 `credential_version` 声明；服务端校验版本不一致时直接判定会话失效，用于覆盖多标签页和旧设备。
- 邮件投递供应商固定为腾讯云邮件推送 HTTP API；本地开发默认使用 `mock` provider，禁止误发真实邮件。
