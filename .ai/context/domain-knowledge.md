# Domain Knowledge

## 交易日历

### A 股与港股的差异

A 股和港股交易日不完全一致，常见差异场景：

1. **休市日不同**: A 股春节休市与港股农历新年休市日期可能错开。
2. **台风休市**: 港股受台风影响可能临时休市，A 股不受影响。
3. **补班日**: 中国内地有调休补班日（非交易日），香港无此概念。

**工程影响**: 任何涉及"前一交易日"的计算，必须按市场维度查询各自的交易日历，不能用"自然日减 1"推算。

### 典型场景

| 日期 | A 股 | 港股 |
|------|-------|-------|
| 交易日当天 20:00 | 优先按当天收盘数据计算；若外部 benchmark 日线尚未确认，则只能回退到最近交易日 | 同理，但必须按中国香港交易日单独判断 |
| 春节假期后首个交易日 | 前一交易日 = 假期前一交易日 | 可能与 A 股不同 |
| 港股台风休市日 | 正常交易 | 休市，前一交易日 = 前一个港股交易日 |

## 收盘后离线计算的时间语义

- **计算触发时间**: 默认采用交易日收盘后的北京时间 20:00，以便让 `computed_at` 与 `source_trade_date` 尽量保持在同一天语义。
- **市场维度跳过**: 自动任务必须按 A 股 / 中国香港各自交易日历判断；某一市场休市时，只跳过该市场，不能连带跳过另一市场。
- **外部数据优先**: 当内部 closing snapshot 尚未稳定落盘时，应优先使用外部 benchmark / 日线数据确认当天是否已有收盘口径，再决定 `source_trade_date` 或自动任务是否跳过。
- **用户展示日期**: 必须展示数据所基于的收盘交易日（`source_trade_date`），而非计算完成时间（`computed_at`）。

**关键认知**: 20:00 调度的目标不是简单“提早运行”，而是尽量让用户看到的 `source_trade_date` 与自动计算发生日保持一致；一旦外部数据仍未刷新，只能显式回退到最近交易日，不能伪装成当天数据。


## 登录与账号安全
- 公网邮件深链统一使用 `APP_PUBLIC_BASE_URL` 生成，当前生产域名为 `https://wolongtrader.top`。
- `MAIL_PROVIDER` 当前支持 `mock` 与腾讯云邮件推送实现，生产建议使用 `tencent`。
- 使用 Docker Compose 启动时，根目录 `.env` 不能只依赖变量插值；`backend` 服务必须通过 `env_file: .env` 或显式 `environment` 映射，未声明的变量否则不会进入容器。
- 找回密码默认参数：token 30 分钟过期、同邮箱 60 秒冷却、同 IP 每小时 10 次、同邮箱每小时 3 次。

## 因子实验室行业口径
- 因子实验室、个股静态资料等用户可见“行业”统一展示 `company_profiles.industry_name`，口径固定为“申万一级行业”。
- `factor_security_industries` 仅作为原始来源捕获表，不直接作为用户展示主口径。
- 港股行业统一记为 `not_applicable`，前端展示为“不适用”，不要为港股强行套用申万行业。
- 行业刷新属于 Factor Lab Phase 0，默认仍在每日 21:00 的自动预计算链路中执行，但外部源失败时只记录 `partial/warning`，不能再阻断 Phase1/Phase2。
- 行业自动源顺序固定为 `akshare -> eastmoney -> tencent`；如果运维显式指定单一源，仍允许只跑该源用于排障。

## 因子实验室刷新频率
- `securities`、`industries`、`daily-bars`、`index-bars`、`financials` 属于夜间自动链路；默认北京时间 21:00 触发 Phase 0，随后串行进入 Phase1/Phase2。
- `dividends` 不再属于夜间自动链路。默认策略是保留最近一次成功结果，并在 admin 后台手动触发全量刷新或缺口修复。
- `dividends` 的手动全量刷新建议窗口：年报/分红实施密集披露后、半年报密集披露后、分红预案集中期、admin 覆盖率或质量告警出现时。
